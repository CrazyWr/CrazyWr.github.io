Network - HTTP 历史及各版本对比

Aug 10, 2017 | Network | 阅读 | 1,091 字 | 4 分钟

HTTP 是基于 TCP/IP 协议的应用层协议. 不涉及数据包 package 传输, 主要规定客户端和服务器之间的通信格式, 默认使用 80 端口.

HTTP/0.9

1991年发布, 极其简单, 只有 GET 方法, 返回数据只能是 HTML 字符串

HTTP/1.0

1996年发布

  • 支持多种格式传输, 包括 文字, 图像, 视频, 二进制文件等等.
  • 支持多种请求方法, 除了 之前的 GET 请求, 添加了 POST, HEAD 请求.
  • 请求和响应的格式改变, 每次通信都包括 头信息 和 数据部分.
    “头信息 + 一个空行（\r\n） + 数据”
    第一行是”协议版本 + 状态码（status code） + 状态描述”
  • 状态码 (status code)
  • 多字符集支持, 头部信息必须是 ASCII 码.
  • 多部分发送 (multi-part type)
  • 权限 (authorization)
  • 缓存 (cache)
  • 内容编码 (content encoding) gzip, compress, deflate …

1.0 版本缺点:

每个 TCP 连接只能发送一个请求, 发送数据完毕, 连接就关闭了, 请求其他资源必须重新建立连接.
建立新 TCP 连接的成本较高 (客户端和服务器三次握手), 外部访问资源比较多时, 性能会变差.

TCP 复用

每个请求使用非标准字段 Connection: keep-alive
这个字段要求服务器不关闭 TCP 连接, 以便其他请求复用, 同时服务器也要回应这个字段, 这样户建立一个复用的 TCP 连接, 直到客户端或服务器一端主动关闭连接 因为不是标准字段, 不同浏览器的实现可能有差别.

HTTP/1.1

1997年1月发布.

  • 默认使用长连接: TCP 连接默认不关闭, 可以被多个请求复用, 不用声明 “Connection: keep-alive”, 客户端发送 “Connection: close”, 明确要求服务器关闭 TCP 连接;
  • 引入管道机制或者说流水线机制: 在同一个 TCP 连接里面, 客户端可以同时发送多个请求.
  • 添加其他方法: PUT, PATCH, HEAD, OPTION, DELETE.
  • 请求头: Host: xxxx.xxxx,
  • 分块传输编码: “Transfer-Encoding: chunked” 每个非空的数据块之前，会有一个16进制的数值，表示这个块的长度。最后是一个大小为0的块，就表示本次回应的数据发送完了

1.1 缺点

虽然 1.1 版本允许复用 TCP 连接, 但是一个 TCP 连接中, 所有的数据通信是按次序返回响应的, 服务器只能响应完前一个请求, 才能响应下一个, 前面的请求堵塞了后面的请求, 造成 头部堵塞

为了避免 头部堵塞, 1. 减少请求书 2. 同事多开持久连接. 对应多种网页优化技巧: 合并脚本, 样式表, 图片并入 CSS 代码, 域名分片等等.

HTTP/2

2009 年谷歌发布 SPDY 协议, 解决 HTTP/1.1 效率不高的问题. 最终多数特性在 HTTP/2 版本中得到继承. 2015年 HTTP/2 发布, 不是2.0, 因为不会发布 2.x x 的子版本.

  • 二进制协议, 1.1 头部信息是 ASCII 编码, 2版本头信息和数据体都是二进制, 统称为 ‘帧’, 可以定义额外的帧, 为将来的高级应用打好了基础.
  • 数据流: 对每个请求进行进行标记, 用来区分属于哪个数据流, 规定客户端发出的数据流 ID为奇数, 服务器发憷的数据流 ID为偶数.
  • 多工: 复用 TCP 连接, 不会再按次序响应请求, 先完成的先响应.
  • 头信息压缩: HTTP 协议不带有状态，每次请求都必须附上所有信息。所以，请求的很多字段都是重复的, HTTP/2 对这一点做了优化，引入了头信息压缩机制（header compression）。一方面，头信息使用gzip或compress压缩后再发送；另一方面，客户端和服务器同时维护一张头信息表，所有字段都会存入这个表，生成一个索引号，以后就不发送同样字段了，只发送索引号
  • 服务器推送: 允许服务器未经请求, 主动向客户端发送资源.
